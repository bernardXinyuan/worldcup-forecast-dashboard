# 多 Agent 协作工作流

## 1. 协作原则

本项目需要多个 agent 分工协作。每个 agent 只负责明确边界内的工作，产出必须写入本地文件，并在交接文档中更新状态。

关键原则：

- 文档是项目事实源。
- 代码不是唯一交付物，数据、预测、部署、验收、运营同等重要。
- 每个阶段结束必须有可检查产物。
- 不依赖单个上下文窗口保存长期记忆。

## 2. Agent 角色定义

### PM Agent

职责：

- 维护项目目标、PRD、范围和优先级。
- 拆分版本路线图和 Backlog。
- 管控需求变更。
- 定义验收标准。

主要文档：

- `docs/00-project-plan.md`
- `docs/01-product-prd.md`
- `docs/05-roadmap-backlog.md`

### UX Agent

职责：

- 设计信息架构、核心页面流程和响应式布局。
- 输出页面结构、交互状态和移动端规则。
- 确保首页、模拟器、路径图和详情页体验清晰。

主要产物：

- 页面原型说明。
- 视觉规范。
- 关键状态清单。

### Data Agent

职责：

- 定义赛事、球队、预测、内容和互动数据模型。
- 评估数据源可用性和授权风险。
- 设计数据更新流程和异常兜底。

主要产物：

- 数据字典。
- 数据更新策略。
- 数据源评估。

### Frontend Agent

职责：

- 实现 Web 看板、模拟器、路径图、球队详情和分享页面。
- 接入配置和数据。
- 完成前端性能、SEO、可访问性和响应式要求。

主要产物：

- 前端应用代码。
- 组件说明。
- 页面截图和测试记录。

### Backend Agent

职责：

- 实现 API、内容配置、投票服务和数据更新任务。
- 提供缓存、鉴权和日志能力。
- 支持预测数据、内容数据和互动数据管理。

主要产物：

- API 文档。
- 服务代码。
- 数据库迁移和种子数据。

### Prediction Agent

职责：

- 设计预测算法或规则模型。
- 输出冠军概率、单场概率、出线概率和路径模拟。
- 提供模型解释和置信度。

主要产物：

- 模型说明。
- 模拟算法。
- 回测或合理性检查记录。

### QA Agent

职责：

- 编写测试计划。
- 验证功能、兼容性、异常状态、移动端表现和分享链路。
- 维护验收清单。

主要产物：

- 测试用例。
- Bug 清单。
- 发布验收报告。

### DevOps Agent

职责：

- 设计部署架构。
- 配置环境、构建、发布、监控和回滚。
- 编写线上部署手册。

主要文档：

- `docs/04-deployment-release.md`

### Content Ops Agent

职责：

- 维护首页推荐、专题文案、热门球队和赛中内容节奏。
- 制定赛前、赛中、赛后运营计划。
- 收集用户反馈和内容表现。

主要产物：

- 内容运营日历。
- 首页推荐计划。
- 赛事复盘模板。

## 3. 标准工作流

### 阶段 1: 需求冻结

负责人：PM Agent

输入：

- 项目想法。
- 目标用户。
- 上线时间要求。

输出：

- PRD。
- MVP 范围。
- 验收标准。

### 阶段 2: 体验和数据设计

负责人：UX Agent、Data Agent、Prediction Agent

输出：

- 页面流程。
- 数据模型。
- 预测输出结构。
- 异常状态设计。

### 阶段 3: 工程初始化

负责人：Frontend Agent、Backend Agent、DevOps Agent

输出：

- 应用脚手架。
- API 契约。
- 本地运行方式。
- 示例数据。

### 阶段 4: MVP 实现

负责人：Frontend Agent、Backend Agent、Prediction Agent

输出：

- 可运行看板。
- 小组模拟器。
- 淘汰赛路径图。
- 球队详情。
- 分享能力。

### 阶段 5: 测试和内测

负责人：QA Agent、Content Ops Agent、DevOps Agent

输出：

- 测试报告。
- 内测链接。
- 用户反馈。
- 发布阻断项清单。

### 阶段 6: 正式发布和赛中运营

负责人：DevOps Agent、Content Ops Agent、PM Agent

输出：

- 生产发布。
- 监控告警。
- SEO 和分享配置。
- 内容运营日历。
- 版本迭代记录。

## 4. 任务交接格式

每个 agent 完成任务后必须更新 `docs/03-context-handoff.md`，格式如下：

```md
## YYYY-MM-DD HH:mm Agent 名称

### 本轮目标

### 已完成

### 修改文件

### 关键决策

### 待处理问题

### 建议下一步
```

## 5. 上下文窗口限制处理

为避免上下文窗口耗尽导致信息丢失：

- 长期事实写入文档，不写在聊天里。
- 每轮只加载当前任务相关文档。
- 大型需求变更要先更新 PRD，再改代码。
- Agent 开始工作前先读 README 和交接文档。
- Agent 结束工作前必须写交接摘要。
- 大型文档按主题拆分，不把所有内容堆进一个文件。

## 6. 冲突处理

当文档、代码和聊天内容冲突时，优先级如下：

1. 用户最新明确指令。
2. 当前 PRD 和项目总规划。
3. 最新交接记录。
4. 代码现状。
5. 旧聊天内容。
